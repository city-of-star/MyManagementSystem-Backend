# WebSocket 企业级封装演进清单（建议按优先级落地）

> 本文目标：告诉你“后面需要做哪些封装”，并对标企业级项目/优秀开源/大厂常见做法。

---

## P0：能稳定上线的必备项

- **统一握手鉴权策略**
  - **网关透传**优先：网关完整校验 JWT + 透传身份头，下游握手只读取头并绑定 `Principal`
  - JWT 直连作为补充：支持 `Authorization`/query param/`Sec-WebSocket-Protocol` 传 token
  - **握手失败处理**：统一 401/403、记录审计日志（user/ip/origin/path）

- **连接生命周期管理**
  - 连接建立/关闭事件：埋点 + 清理注册表
  - 每用户连接数限制（如：1~3），防止刷连接
  - 空闲超时踢下线（idle timeout）

- **心跳与保活**
  - 应用层 `ping/pong`（文本或二进制）
  - 统计 RTT、断线重连策略建议（客户端）

- **限流与防护**
  - 按 userKey/ip 维度限流（握手频率、消息频率）
  - 消息大小限制、帧数限制（防止 OOM）
  - Origin 校验（避免被第三方网页滥连）

- **可观测性**
  - 指标：连接数、在线用户数、入/出消息量、失败数、平均延迟、慢消费者数量
  - 日志：traceId 贯穿（握手从 `X-Trace-Id` 注入 MDC）

## P1：大多数企业项目都会要的增强

- **统一消息协议（强烈建议）**
  - 字段建议：`messageId`、`type`、`timestamp`、`data`、`traceId`
  - 支持 `ack`（确认）、`nack`（失败原因）、`seq`（顺序）
  - 幂等：按 `messageId` 去重（尤其是重连/重试场景）

- **服务端推送能力完善**
  - `sendToUser(userId)`、`sendToUsers(batch)`、`sendToRoom(roomId)`、`broadcast()`
  - 离线消息（可选）：写入 Redis/DB，用户上线拉取

- **权限模型**
  - 订阅/加入房间时做鉴权：角色/权限/资源维度
  - 权限动态变更：踢下线、撤销订阅、降低推送频率

- **异常与降级**
  - 处理慢消费者：队列上限、丢弃策略、断开策略
  - 服务繁忙时：拒绝新连接、拒绝高频消息、按优先级降级

## P2：集群/大厂级实时系统能力（关键在“跨节点”）

### 2.1 架构选择（常见 4 种）

- **A. Sticky Session（简单）**
  - 负载均衡按 userId 维度一致性路由到同一台实例
  - 优点：实现成本低
  - 缺点：弹性扩缩容、故障切换复杂；热点用户会打爆单点

- **B. 边缘层专用 WS 网关（大厂常见）**
  - WS 连接都在“实时网关/边缘服务”，后端业务通过 MQ/HTTP 指令推送
  - 优点：后端无连接状态，扩缩容简单
  - 缺点：需要搭一套实时网关与路由通道

- **C. Redis/Kafka 路由（常用）**
  - 每个节点维护本地连接；全局推送先确定目标节点，再通过 Redis PubSub/Kafka 通知到节点发给客户端
  - 关键点：维护 `userId -> nodeId` 映射（Redis），节点心跳续约

- **D. STOMP + Broker Relay（Spring 生态最成熟）**
  - Spring `spring-messaging` + STOMP，路由交给 Broker（RabbitMQ/ActiveMQ）
  - 优点：开发体验好（topic/user destination）
  - 缺点：体系更重，性能与运维成本更高

### 2.2 大厂通常会补的能力

- **Presence/在线状态服务**：统一维护在线/离线、最后活跃时间、多端信息
- **房间/群组路由服务**：大房间拆分、分片、扇出优化（fan-out）
- **消息可靠性**：至少一次/至多一次/恰好一次的取舍；客户端补偿
- **审计与合规**：关键消息落库、可追溯、敏感词/内容安全（如 IM 场景）

## 对标优秀开源/企业实践：他们通常怎么做

- **Spring 官方路线**
  - 小规模/中规模：`spring-websocket`（原生 WebSocket）+ 自己写 handler/registry
  - 需要“订阅/广播/用户路由”：`spring-messaging` + STOMP（`SimpMessagingTemplate`、user destination）

- **Netty 路线（高性能/强定制）**
  - 自定义协议、二进制帧、背压控制、连接管理更强
  - 常见于：IM、行情推送、游戏、IoT 等高并发实时场景

- **大厂常见路线（工程化优先）**
  - WS 连接前置到边缘层（网关/专用 realtime 服务）
  - 后端业务通过 MQ 或者内部 RPC 触发推送
  - 全链路监控与安全策略（限流、风控、黑名单）
