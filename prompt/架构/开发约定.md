# 开发约定

## 1. 安全与身份规则

- `SecurityContext` 仅存最小认证信息，不存完整用户资料。
- 请求身份字段（`userId`、`username`）优先由 `UserUtils` 获取，不在usercenter服务再用`UserContextUtils`获取，需要详细信息调用usercenter服务的feign接口
- 角色 authority 必须使用 `ROLE_` 前缀。
- 权限校验优先复用 `@RequiresPermission` + `PermissionCheckAspect`。

## 2. 异常与响应规则

- 业务失败优先使用 `BusinessException` + 统一错误码。
- 非预期系统异常可使用 `ServerException` 并保留根因日志。
- API 返回遵循项目统一响应模型。

## 3. 分层职责规则

- `controller`：参数接收与响应编排。
- `service`：业务编排与事务边界。
- `mapper`：持久层访问。
- 跨服务通用能力优先放入 `mms-common-bc`。

## 4. 变更一致性规则

- 鉴权链路变更必须校验网关/服务侧请求头契约一致。
- 角色格式变更必须端到端验证：写入格式、读取格式、权限判断。
- 新增透传请求头优先在 `GatewayConstants.Headers` 统一定义。

## 5. Prompt 文档维护规则

- 新的高频排障结论应补充到 prompt 文档。
- 架构或安全链路变更后，优先更新：
  - `架构/项目总览.md`
  - `安全/鉴权链路.md`

## 6. 代码风格与实现习惯（后端）

- 分层职责保持稳定：`controller` 仅做参数接收与响应编排，`service` 负责业务与事务，`mapper` 负责持久化访问。
- 命名遵循现有后缀约定：`XxxController`、`XxxService`、`XxxServiceImpl`、`XxxMapper`、`XxxDto/Vo/Entity`。
- Controller 统一返回 `Response<T>`，成功优先使用 `Response.success(...)`。
- 参数校验优先前置到入参 DTO：`@RequestBody @Valid` + `jakarta.validation` 注解。
- 写操作默认加 `@Transactional(rollbackFor = Exception.class)`，事务边界放在 service 层。
- 异常处理保持分层：业务分支抛 `BusinessException`，非预期异常记录日志后抛 `ServerException`。
- 权限校验优先复用 `@RequiresPermission`，权限编码优先复用常量定义，避免字符串散落。
- 身份信息读取优先 `UserContextUtils`，`SecurityContext` 仅保留最小认证信息与 authority 视图。
- 对象转换沿用既有风格：手写转换为主，必要时使用 `BeanUtils.copyProperties`，避免无约束复制。
- 日志保持 `@Slf4j` + 占位符风格，按 `info/warn/error` 分级记录关键业务事件。
- 新增类、接口、枚举等类型定义时，必须添加文件头注释（含“实现功能”描述、`@author`、`@date`），格式参照 `mms-usercenter-bc-service/src/main/java/com/mms/usercenter/service/security/service/impl/OnlineUserServiceImpl.java`，其中 `@date` 必须填写创建当下的当前时间。
- 设计表的时候，需要做两件事，一个是直接修改 `mysql/init_mms_dev_core.sql` 这个文件，这个文件用于一次性初始化的，不需要更新表结构什么的，一个是新建变更sql文件，文件名参考 "20260331_1705_alter_system_user_avatar.sql"
